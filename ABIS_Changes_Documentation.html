Project Structure — How All Files Connect

Your project follows a standard Java Servlet + JSP architecture. Here is every file and what it does:

File PathTypePurpose
src/main/webapp/WEB-INF/web.xmlXMLMaps URLs to Servlet classes. E.g. /app/enrollment → EnrollmentServlet
src/main/webapp/WEB-INF/jsp/main.jspJSPMaster layout — includes all fragment files
WEB-INF/jsp/fragments/footer.jspJSPContains the Payload Overlay HTML — the popup that shows REQUEST JSON + SUMMARY + RESPONSE

💡
Key rule: web.xml is the router Every URL your browser visits goes through web.xml first. The browser never talks directly to a Java class — it hits a URL, web.xml maps it to the correct Servlet, and the Servlet handles it.

Problem 1 — Wrong POST Method (PRG Pattern vs AJAX/Fetch)

🔴
Root Cause: The original servlet used PRG (Post-Redirect-Get) pattern — incompatible with JS fetch() The original doPost() did a server-side redirect. Fetch() cannot follow redirects to show JSON — it receives HTML back instead.

Original EnrollmentServlet.java — What Was Wrong

EnrollmentServlet.java — doPost() — ORIGINAL (BROKEN)
// No @MultipartConfig annotation — cannot read file uploads
// No Jackson imports — cannot parse JSON
// Uses request.getParameter() for ALL data (assumes HTML form)
applicant.setFirstName(request.getParameter("firstName")); // ❌ data not here
BiometricData biometrics = new BiometricData(); // ❌ empty — no files read
response.sendRedirect(request.getContextPath() + "/app/enrollment"); // ❌ redirect!
⚠️
Why This Fails When JS does fetch("enrollment", { method: "POST", body: formData }) and the server responds with a redirect (HTTP 302), the browser follows the redirect automatically and returns the HTML page — not JSON. JS then tries to call .json() on HTML text and crashes with a parse error.

What Was Changed — New doPost() Approach

EnrollmentServlet.java — doPost() — FIXED
@MultipartConfig(maxFileSize = 2*1024*1024, maxRequestSize = 30*1024*1024) // ✅ Enable file upload
response.setContentType("application/json"); // ✅ Always return JSON
response.setCharacterEncoding("UTF-8");
String jsonStr = request.getParameter("json"); // ✅ Read the "json" text part
JsonNode root = objectMapper.readTree(jsonStr); // ✅ Parse it with Jackson
JsonNode demo = root.path("demographic"); // ✅ Get the demographic block
applicant.setFirstName(textOf(demo, "firstName")); // ✅ Extract each field
biometrics.setFaceFront(readFilePart(request, "faceFront")); // ✅ Read image files
response.getWriter().write("{\"status\":\"success\",...}"); // ✅ Return JSON, NOT redirect

Why FormData is Sent This Way

The JS sends data as multipart/form-data with two types of parts:

Part NameTypeContentHow Servlet Reads It
jsonTextFull JSON payload string with demographic datarequest.getParameter("json")
faceFrontFilePNG/JPG/BMP image bytesrequest.getPart("faceFront")
fp_left_thumbFileFingerprint image bytesrequest.getPart("fp_left_thumb")
irisLeftFileIris image bytesrequest.getPart("irisLeft")

Problem 2 — Raw Response Object Not Parsed (.json() Missing)

🔴
Root Cause: fetch() returns a Response object, not data. You must call .json() to parse it. The first version of the fetch code passed the raw Response object directly to displayPayload. This is why the console showed Response { json: f json(), ok: (...), status: (...) } instead of { status: "success", message: "..." }
main.js — handleSubmit() fetch chain
// ORIGINAL — BROKEN: .then(data => ...) receives raw Response, not parsed JSON
fetch("enrollment", { method: "POST", body: formData })
.then(data => {
console.log(data); // ❌ prints Response object, NOT your JSON
displayPayload(lastPayload, data); // ❌ data.status is undefined
})
 
// FIXED: Two .then() calls — first parses bytes to JSON, second uses it
fetch("enrollment", { method: "POST", body: formData })
.then(res => res.json()) // ✅ Step 1: parse Response bytes → JS object
.then(data => { // ✅ Step 2: now data = { status: "success", message: "..." }
lastResponse = data;
showToast(data.message);
displayPayload(lastPayload, data);
})
💡
Why Two .then() Calls Are Needed fetch() is asynchronous at two levels. First level: waiting for bytes to arrive from server. Second level: waiting for those bytes to be parsed as JSON. Each level needs its own .then(). This is standard fetch API design.

Problem 3 — Overlay Used Wrong Method to Show

🔴
Root Cause: displayPayload used overlay.style.display = 'block' but the CSS animation relies on classList 'visible' Your CSS uses .payload-overlay.visible { ... } to control visibility. Setting style.display directly bypasses the CSS class system, so animations break and closePayload() fails because it checks classList.contains('visible') which is never true.
main.js — displayPayload() — overlay section
// BROKEN: bypasses CSS class system
overlay.style.display = 'block';
 
// FIXED: uses classList — matches how closePayload() checks
overlay.classList.remove('closing');
overlay.classList.add('visible');

The closePayload() function was written to check overlay.classList.contains('visible'). If visible was never added as a class, close would work once but re-opening would fail silently.

Problem 4 — Wrong HTML Element IDs in JS

🔴
Root Cause: JS used IDs that did not exist in the HTML. The HTML used different IDs. Debug log showed: respBlock found: false, respSection found: false. The JS was looking for getElementById('responseBlock') and getElementById('responseSection') — neither of which existed in footer.jsp.

What the JS Was Looking For vs What Actually Existed in footer.jsp

JS Was UsingActual ID in footer.jspFix Applied
responseBlockresponseCardBodyChanged JS to use responseCardBody
responseSectionNo ID — has class response-sectionUsed .closest('.response-section') to find it
.response-waiting.response-loader wrapping .response-loader-textUsed broader selector to find loader
main.js — displayPayload() — response element lookup
// BROKEN: IDs that don't exist in footer.jsp
const respBlock = document.getElementById('responseBlock');
const respSection = document.getElementById('responseSection');
 
// FIXED: use the real ID from footer.jsp
const respBlock = document.getElementById('responseCardBody');
const respSection = respBlock ? respBlock.closest('.response-section') : null;
💡
How to Find Real IDs in Future Right-click any element in browser → Inspect → look at id="..." attribute. Or run in console: document.querySelectorAll('[id]').forEach(el => console.log(el.id))

Problem 5 — Ternary Bug in displayPayload

⚠️
Minor Bug: Assignment inside ternary operator — works but is wrong and confusing
main.js — displayPayload() — JSON rendering
// BUGGY: assignment as fallback side-effect
jsonBlock.innerHTML = typeof syntaxHighlight === "function"
? syntaxHighlight(jsonStr)
: jsonBlock.innerText = jsonStr; // ← assignment INSIDE ternary
 
// CORRECT: clean if/else
if (typeof syntaxHighlight === "function") {
jsonBlock.innerHTML = syntaxHighlight(jsonStr);
} else {
jsonBlock.innerText = jsonStr;
}

The same pattern existed for respBlock rendering. Both should be converted to clean if/else blocks.

All Backend (Java) Changes Made

EnrollmentServlet.java — Complete List of Changes

#What ChangedWhy
1 Added @MultipartConfig annotation with size limits Without this, request.getPart() throws exception. Servlet cannot read file uploads without it.
2 Added Jackson imports: ObjectMapper, JsonNode Jackson is the JSON parsing library. Needed to parse the "json" text part into Java objects.
3 Added Part, InputStream, Base64 imports Needed to read uploaded image files and convert to Base64 strings.
4 doPost() — removed all request.getParameter() for demographic fields Old method expected HTML form fields. New method reads them from the JSON string instead.
5 doPost() — added request.getParameter("json") + Jackson parse JS sends everything as one JSON string in the "json" part. We parse that to get demographic data.
6 doPost() — replaced response.sendRedirect() with JSON write Redirect breaks fetch(). Must return JSON directly so JS can read it.
7 doPost() — added response.setContentType("application/json") Browser must know the response is JSON so it can parse it with .json().
8 Added readFilePart() helper method Reads one image file part → converts bytes to Base64 → returns CaptureSlot object.
9 Added extractFilename() helper method Parses Content-Disposition header to get original filename from uploaded file.
10 Added textOf() helper method Null-safe JSON field extraction. Returns null if field missing — prevents NullPointerException.
11 Added sendError() helper method Returns HTTP 400 with JSON error body instead of server crash page.
12 Removed flash message handling from GET (enrollmentSuccess/enrollmentError) Not needed anymore — responses go directly via JSON, not via session flash.

EnrollmentService.java — No Changes Needed

The service was already correct. It validates applicantId and biometrics.getFilledCount() > 0, then returns a success message string. The BiometricData model correctly counts filled CaptureSlots.

BiometricData.java — No Changes Needed

The model was already set up correctly with CaptureSlot inner class. isFilled() checks if base64Data is not null/empty. The new servlet correctly calls setFaceFront(), setLeftIris() etc.

All Frontend (JS) Changes Made

handleSubmit() — Complete Rewrite

#What ChangedWhy
1 Added lastPayload = payload before fetch Without this, displayPayload(lastPayload, data) passes null as first argument.
2 Replaced direct form submit with fetch() + FormData Allows AJAX-style submission that stays on page and receives JSON response.
3 Added formData.append("json", JSON.stringify(payload)) Sends demographic data and metadata as a JSON string the servlet can parse.
4 Added file appends for face/iris/fingerprints Each uploaded image file is appended separately as a multipart part.
5 Added .then(res => res.json()) as first then() This is the critical fix — parses raw Response bytes into a JS object.
6 Added lastResponse = data in second then() Needed so copyResponseJSON() can access the last server response.

displayPayload() — Changes Made

#What ChangedWhy
1 Changed signature from displayPayload(payload) to displayPayload(payload, backendResponse) Original only showed request JSON. Now also shows the server response.
2 Changed getElementById('responseBlock') → getElementById('responseCardBody') responseBlock did not exist in footer.jsp. responseCardBody is the real ID.
3 Changed getElementById('responseSection') → closest('.response-section') No ID existed on the parent div — it only had a class name.
4 Changed overlay.style.display = 'block' → overlay.classList.add('visible') CSS animation and closePayload() rely on the 'visible' class, not inline style.
5 Added null checks on all element lookups Prevents "Cannot read properties of null" errors if any element is missing.

copyResponseJSON() — Changed

main.js — copyResponseJSON()
// ORIGINAL: always showed "No response data available"
function copyResponseJSON() {
showToast('No response data available');
}
 
// FIXED: uses lastResponse set by fetch() handler
function copyResponseJSON() {
if (!lastResponse) { showToast('No response data available'); return; }
const text = JSON.stringify(lastResponse, null, 2);
navigator.clipboard.writeText(text).then(() => showCopyFeedback(btn));
}
